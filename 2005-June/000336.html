I'm against a lisp scripting language as it is for the normal user much
more complicated to learn than an object oriented programming language.
My favourite is ruby. Concerning Lilypond: If a user is able to make 
changes in his lilypond score that touch the lisp interface of lilypond
than he has completely different abilities than a normal NoteEdit user
has. Many of the possible NoteEdit users I know would ever dare to touch
a script written in a lisp language.

The first step for an advanced API is to actually document the API. The
base is of course the doxygen documentation and with that I would have
to find out where the interface between the script and NoteEdit has to
be drawn. The NoteEdit library is too big as an API for the scripting 
language itself (actually it contains complete NoteEdit, I ask myself
why the library is so big). What could be inside the API ?

- control of all typesetting elements
- control of staff / voices
- editing control
- file control
- Midi interface control
- import/export control

Things like transposition, change clef etc. should be done via scripts
or macros in the future. Also my idea is to provide many default macros 
to the user as a base for composition work and for own scripts/macros.

There probably is no real difference between macros and scripts. I would
call a macro a short script which can be recorded and played back within
noteedit. This would be another way to create a basic script that can
later be expanded with your favourite text editor :).

Currently I can't really imagine why NoteEdit requires a real plugin
support. I think scripting / macros support is enough. It still probably
will be painful enough to expand NoteEdit so that a scripting interface
can be created.
